- Prefer kebab-case for all TS/JS files.
- Streaming runs through the vendored scrcpy server (`vendor/scrcpy-server-v<VERSION>`). The version is pinned in `src/scrcpy-server.ts`; the wire protocol drifts between scrcpy majors, so bumping the version means re-validating `src/scrcpy.ts` against the new server.
- Server protocol summary (scrcpy 4.x, `send_stream_meta`): open two sockets through `adb forward tcp:<port> localabstract:scrcpy_<scid>`; both prefix a 1-byte dummy. Video socket then yields 64-byte device name, 16-byte stream meta (codec_id, session flags, width, height BE), then packets of `[8B PTS BE, 4B size BE, N B Annex-B NALUs]` where the PTS top bits mark session/config/keyframe (bit 63/62/61). A "session" packet (bit 63) carries a new width/height instead of frame data — the encoder restarted, e.g. on device rotation. `src/scrcpy.ts` also still parses the scrcpy 3.x preamble/flags (12-byte codec meta; config bit 63, keyframe bit 62). Control socket consumes binary messages described in `src/input.ts`.
- Default device target: the only booted device. If multiple, require `-s <serial>`.
- Don't shell out to `adb shell input` — write directly to scrcpy's control socket; round-trip latency is ~5ms vs ~200ms.
